# LIVE LIFE 追加文档建议审批

状态：待审批  
最后更新：2026-06-08

当前已经整理出：

- 需求分析。
- 后端详细设计。
- 前端多版本设计语言。
- 本地开发说明。
- 部署文档。
- V2/V3 阶段审批稿。

下面是我建议后续补的文档。它们不是现在必须全部写完，但一旦项目继续推进，会明显减少沟通成本。

## 1. 内容录入规范

建议文件名：

```text
content-data-entry-guide.md
```

用途：

- 规定演出、CD、黑胶、Archive 的字段怎么填。
- 规定三语言内容缺失时如何处理。
- 规定图片命名、尺寸、版权备注、外部链接如何记录。

建议优先级：高。

原因：

- 之后你会不断给活动图、CD 信息、文本内容。
- 如果没有录入规范，前端和后端会频繁被内容格式打断。

## 2. 外部链接与购买路径规范

建议文件名：

```text
external-link-policy.md
```

用途：

- 规定 BASE、票站、Instagram、小红书等外部链接怎么展示。
- 规定按钮文案，例如 `点击此处购买`。
- 规定什么时候需要写“外部 Shop”提示。
- 避免用户误以为 LIVE LIFE 站内已经完成支付或出票。

建议优先级：高。

## 3. Review 模式说明

建议文件名：

```text
review-mode-guide.md
```

用途：

- 说明右上角版本下拉框怎么工作。
- 说明 `?design=v2`、`?design=v3` 这类 URL 参数。
- 说明哪些控件只在客户 Review 环境出现。
- 说明正式上线时默认展示哪个版本。

建议优先级：中高。

原因：

- 之后会有多版设计同时存在，必须避免 Review 控件误进正式页面。

## 4. 品牌与文案规范

建议文件名：

```text
brand-language-guide.md
```

用途：

- 固定 `LIVE LIFE` 大写规则。
- 固定三语言导航翻译。
- 固定英文全大写范围。
- 固定经典乐队名纹理的免责声明边界。
- 规定如何避免“合作/授权/出演”误读。

建议优先级：中高。

## 5. 登录注册与站内支付备忘

建议文件名：

```text
future-auth-checkout-notes.md
```

用途：

- 等决定是否做登录注册后使用。
- 记录账户、订单、支付、发货、售后所需模块。
- 对比外部 Shop 与站内支付的成本。

建议优先级：中。

当前不急，因为购买路径已经先定为外部 Shop。

## 6. 数据库设计草案

建议文件名：

```text
database-schema-draft.md
```

用途：

- 把当前 Go seed 数据转换成未来 SQLite 表结构。
- 规划 events、catalog_items、contents、connect_messages 等表。
- 为部署到服务器后的持久化做准备。

状态：已生成第一版，见 `docs/database-schema-draft.md`。

## 7. 图片资产规范

建议文件名：

```text
image-asset-guide.md
```

用途：

- 规定首页图、演出图、CD 图、Archive 图放哪里。
- 规定尺寸、压缩、命名、替代文本。
- 规定哪些图可以公开展示，哪些只是内部参考。

建议优先级：中。

## 审批建议

我建议下一批先补这四个：

1. `content-data-entry-guide.md`
2. `external-link-policy.md`
3. `review-mode-guide.md`
4. `brand-language-guide.md`

原因：

- 它们最贴近你接下来要上传图片、给客户看版本、继续改视觉这几个动作。
- 数据库第一版已经补齐；登录注册可以稍后，等功能真的进入下一阶段再写细。
